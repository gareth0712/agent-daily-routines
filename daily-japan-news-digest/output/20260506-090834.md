チーム開発で Claude Code の Auto Memory を 3 ヶ月運用した経験から導いた 5 つの設計原則。重要な前提として「Auto Memory は機械学習ではなくファイル書き込み自動化」であり、Claude の重みは更新されず、セッションごとに `MEMORY.md` が注入されるだけ。確認された罠は：① 200 行上限による情報押し出し、② CI/CD 環境でのデフォルト有効による不安定化、③ `CLAUDE.md`（人間が書くルール）と Auto Memory（AI が記録する観察）の混同、④ リファクタリング後の陳腐化メモリによる誤誘導、⑤ ローカルメモリの蓄積が異なる開発者間で矛盾を生む。対策は CI 環境では `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` で無効化し、チーム共通ルールはバージョン管理された `CLAUDE.md` に、一時的な発見だけを Auto Memory に分離する設計。定期的（月 1 回推奨）のメモリ棚卸しも必須。
